# 提示保护规则

AxonHub 提供基于正则表达式的提示保护系统，允许您在 API 请求发送到 AI 提供商之前，对敏感内容进行脱敏或拒绝处理。

## 概述

提示保护规则允许您定义正则表达式模式来匹配用户消息中的敏感内容。当匹配成功时，系统可以：

- **脱敏（Mask）** - 使用自定义替换字符串替换匹配的内容
- **拒绝（Reject）** - 阻止整个请求并返回错误

此功能适用于：
- 保护敏感数据（API 密钥、密码、信用卡号）
- 防止提示注入攻击
- 遵守数据隐私法规
- 过滤不当内容

## 核心概念

### 规则结构

每个提示保护规则包含：

| 字段 | 类型 | 描述 |
|------|------|------|
| **名称** | 字符串 | 规则的唯一标识符 |
| **描述** | 字符串 | 规则用途的可选描述 |
| **模式** | 字符串 | 用于匹配内容的正则表达式 |
| **状态** | 枚举 | `enabled`（启用）、`disabled`（禁用）或 `archived`（已归档） |
| **设置** | 对象 | 操作配置和作用范围设置 |

### 设置配置

`settings` 对象包含：

| 字段 | 类型 | 必填 | 描述 |
|------|------|------|------|
| **action** | 枚举 | 是 | `mask`（脱敏）或 `reject`（拒绝） |
| **replacement** | 字符串 | `mask` 操作必填 | 用于替换匹配内容的文本 |
| **scopes** | 数组 | 是 | 规则应用的消色角色 |

### 操作类型

| 操作 | 描述 |
|------|------|
| **mask** | 使用 `replacement` 字符串替换匹配的内容。请求继续执行，但内容已被修改。 |
| **reject** | 完全阻止请求。向客户端返回错误。 |

### 作用范围

作用范围定义规则应用于哪些消息角色：

| 范围 | 描述 |
|------|------|
| **system** | 系统消息（如"你是一个有帮助的助手"） |
| **developer** | 开发者消息（某些 API 格式中使用） |
| **user** | 用户消息 |
| **assistant** | 助手消息（之前的响应） |
| **tool** | 工具结果消息 |

您可以选择多个作用范围。如果未选择任何范围，规则将应用于所有消息类型。

## 工作原理

### 请求流程

```
┌─────────────────┐
│   客户端请求     │
│   包含消息       │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────────────────┐
│          提示保护处理管道                        │
│  ┌──────────────────────────────────────────┐  │
│  │ 1. 加载所有启用的规则（缓存）              │  │
│  │ 2. 对每个规则：                           │  │
│  │    - 检查作用范围是否匹配消息角色          │  │
│  │    - 对内容应用正则表达式匹配              │  │
│  │    - 执行操作（脱敏/拒绝）                 │  │
│  └──────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐     ┌─────────────────┐
│  脱敏后的请求    │ 或  │   错误响应       │
│  （已修改）      │     │  （如被拒绝）    │
└────────┬────────┘     └─────────────────┘
         │
         ▼
┌─────────────────┐
│   AI 提供商      │
│   接收受保护的   │
│   内容           │
└─────────────────┘
```

### 规则处理顺序

规则按 ID 升序处理。当多个规则匹配相同内容时：

1. **脱敏操作**按顺序应用 - 每个规则的替换应用于前一个规则的结果
2. **拒绝操作**立即终止处理 - 第一个拒绝匹配会阻止请求

## 创建规则

### 通过管理界面

1. 在侧边栏导航到 **提示保护规则**
2. 点击 **创建规则**
3. 配置规则：
   - 输入唯一的 **名称**
   - 添加可选的 **描述**
   - 定义 **正则表达式模式**
   - 选择 **操作类型**（脱敏或拒绝）
   - 如果选择脱敏，输入 **替换文本**
   - 选择规则应用的 **作用范围**
4. 点击 **创建**
5. 点击状态切换开关启用规则

### 模式示例

#### API 密钥和令牌

```
# 匹配 OpenAI API 密钥
sk-[a-zA-Z0-9]{48}

# 匹配通用 API 密钥
(?i)api[_-]?key['\"]?\s*[:=]\s*['\"]?[a-zA-Z0-9_-]{20,}

# 匹配 Bearer 令牌
Bearer\s+[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+
```

#### 信用卡号

```
# 匹配 Visa/Mastercard 卡号
\b(?:\d{4}[-\s]?){3}\d{4}\b

# 带上下文匹配
(?i)(credit\s*card|card\s*number)['\"]?\s*[:=]\s*['\"]?\d{13,19}
```

#### 密码和密钥

```
# 匹配密码模式
(?i)password['\"]?\s*[:=]\s*['\"]?[^\s'\"<>]{8,}

# 匹配 AWS 密钥
(?i)aws[_-]?secret[_-]?access[_-]?key['\"]?\s*[:=]\s*['\"]?[a-zA-Z0-9/+=]{40}
```

#### 电子邮件地址

```
# 匹配电子邮件地址
\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b
```

#### 电话号码

```
# 匹配美国电话号码
\b(?:\+?1[-.\s]?)?\(?[0-9]{3}\)?[-.\s]?[0-9]{3}[-.\s]?[0-9]{4}\b
```

## 使用示例

### 示例 1：脱敏 API 密钥

**规则配置：**
- 名称：`mask-api-keys`
- 模式：`sk-[a-zA-Z0-9]{48}`
- 操作：`mask`
- 替换文本：`[REDACTED_API_KEY]`
- 作用范围：`user`、`system`

**处理前：**
```json
{
  "messages": [
    {"role": "user", "content": "My API key is sk-abcdefghijklmnopqrstuvwxyz1234567890ABCDEFGHIJKLMNOPQRSTUVWXYZ"}
  ]
}
```

**处理后：**
```json
{
  "messages": [
    {"role": "user", "content": "My API key is [REDACTED_API_KEY]"}
  ]
}
```

### 示例 2：拒绝敏感数据

**规则配置：**
- 名称：`reject-credit-cards`
- 模式：`\b(?:\d{4}[-\s]?){3}\d{4}\b`
- 操作：`reject`
- 作用范围：`user`

**请求：**
```json
{
  "messages": [
    {"role": "user", "content": "My credit card is 4111-1111-1111-1111"}
  ]
}
```

**响应：**
```json
{
  "error": {
    "message": "Invalid request: request blocked by prompt protection policy",
    "type": "invalid_request_error"
  }
}
```

### 示例 3：特定范围保护

**规则配置：**
- 名称：`protect-system-prompt`
- 模式：`(?i)ignore\s+(previous|all)\s+(instructions|rules)`
- 操作：`reject`
- 作用范围：`user`

此规则仅应用于用户消息，防止提示注入攻击，同时允许系统提示包含类似短语。

## 规则管理

### 状态管理

| 状态 | 描述 |
|------|------|
| **enabled** | 规则已激活并应用于请求 |
| **disabled** | 规则未激活但可以重新启用 |
| **archived** | 规则已软删除并从列表中隐藏 |

### 批量操作

您可以对多个规则执行批量操作：

- **批量启用** - 同时启用多个规则
- **批量禁用** - 同时禁用多个规则
- **批量删除** - 永久删除多个规则

### 缓存行为

启用的规则会被缓存以提高性能：

- 缓存每 30 秒刷新一次
- 变更会触发异步缓存重载
- 模式编译单独缓存

## 最佳实践

### 模式设计

1. **精确匹配** - 避免过于宽泛的模式，可能会匹配到非预期内容
2. **测试模式** - 在部署前使用正则测试工具验证模式
3. **考虑上下文** - 适当时包含上下文词汇以减少误报
4. **使用锚点** - 使用 `\b` 词边界避免部分匹配

### 规则组织

1. **命名规范** - 使用能表明规则用途的描述性名称
2. **顺序重要** - 规则按 ID 处理；对于重叠模式需考虑顺序
3. **范围选择** - 仅选择必要的作用范围以减少处理开销
4. **文档说明** - 添加描述说明规则的用途

### 安全考虑

1. **深度防御** - 对关键数据类型使用多个规则
2. **定期审查** - 定期检查和更新模式
3. **监控匹配** - 检查日志中的规则匹配以识别潜在问题
4. **充分测试** - 在生产环境启用前验证规则是否按预期工作

## 常见问题

### Q: 为什么我的规则没有匹配？

A: 请检查以下几点：
1. 规则是否已 **启用**？
2. **作用范围** 是否匹配消息角色？
3. **正则表达式模式** 是否正确？使用正则工具测试
4. 模式中是否有语法错误？

### Q: 如何测试模式？

A: 使用在线正则测试工具如 [regex101.com](https://regex101.com) 或 [regexr.com](https://regexr.com)。选择 Go 正则流派以获得准确结果。

### Q: 多个规则匹配时会发生什么？

A:
- **所有脱敏规则**按顺序应用
- **第一个拒绝规则**立即阻止请求
- 规则按 ID 升序处理

### Q: 替换中可以使用捕获组吗？

A: 目前替换是静态字符串。不支持捕获组替换。使用整个匹配内容替换的方式。

### Q: 如何处理多行内容？

A: 使用 `(?s)` 标志启用 dot-all 模式，或使用 `[\s\S]` 匹配包括换行符在内的任何字符：

```
(?s)secret:\s*(.*?)(?=---|$)
```
