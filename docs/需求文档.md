# 用户订阅套餐需求文档（融合版）

请实现以下需求。

## [背景引入]

我们需要面向顾客推出一种用户订阅套餐。顾客可以购买套餐后使用套餐额度调用模型。套餐购买支持使用现有余额支付，也可以只走现金支付渠道，或者同时支持余额与现金支付，具体支付方式由管理员在系统设置中配置。

本需求中的“套餐”指用户侧订阅套餐，内部命名建议统一使用 `subscription_plan` / `subscription` 相关命名，避免和现有项目中的“上游套餐渠道 / 上游套餐额度查询”概念混淆。现有 `ChannelInfo.IsPlan`、GLM/Kimi/Minimax 套餐查询等属于上游供应商套餐或渠道套餐，建议内部称为 `provider_plan` / `channel_plan`，不得与用户订阅套餐复用语义。

## [核心业务规则]

1. 用户存在有效订阅套餐时，API 调用优先消耗套餐额度。
2. 套餐额度不足时，直接拒绝调用，不回退扣用户余额。
3. 如果实际消耗超过预扣，且套餐余额不足，允许本次调用产生末次超额；需要记录超额流水，并阻止该用户后续继续使用套餐额度调用。
4. 用户同一时间只能有一个 `active` 状态的订阅套餐。
5. 续订应生成下一个周期的 `pending` 订阅，不影响当前 `active` 订阅。
6. 升级套餐时，用户立即补差价升级，并重开完整新周期。
7. 升级时旧套餐已用额度需要保留并计入新套餐总额度消耗。
8. 如果升级后 `used_total >= new_total_quota`，新套餐应立即显示总额度已耗尽，周期仍有效，但不可继续调用。
9. 套餐刷新周期按订阅生效时间滚动刷新，而不是按自然日、自然月固定刷新。
10. 渠道是否支持套餐用户调用，仅用于筛选可用渠道；如果当前模型没有支持套餐的可用渠道，应返回“无可用渠道”。
11. 套餐购买订单需要新建独立订阅订单，不复用充值订单。
12. 退订当前只做工单引导，不提供实际退订 API；前端按钮文案建议为“申请退订”或“联系客服”，避免用户误解为点击后立即退订成功。

5. 渠道管理的新建/编辑页面新增配置项：该渠道是否支持用户订阅套餐调用。
6. 用户管理中支持给用户发放套餐、变更套餐、移除套餐，以及提前添加下一个周期的套餐。

8. 动态倍率新增配置项：是否对套餐计费生效。

## [各功能模块需求讲解]

### <系统设置-套餐设置>

管理员可以在此处配置套餐支付方式和套餐模型消耗倍率。

#### 1. 支付方式

支付方式使用下拉列表，支持以下值：

- `余额支付`
- `现金支付`
- `余额&现金都可`

这里的现金支付指管理员在支付方式中配置的支付渠道，例如易支付、Stripe 等。套餐订单必须和充值订单区分处理，支付回调不能默认按充值订单完成，否则会把套餐付款错误地当作充值并给用户加余额。

建议实现方式：

- 套餐订单使用独立订阅订单表。
- 订阅订单使用独立 `trade_no` 前缀，或新增明确的 `order_type = subscription`。
- 所有支付回调入口需要统一分发订单类型。
- 充值订单完成后走充值逻辑；套餐订单完成后走订阅开通、续订或升级逻辑。

#### 2. 模型消耗倍率

套餐模型消耗倍率参考现有价格设置交互，使用表格配置，但这里只配置模型倍率一个值，类似分组倍率。默认倍率为 `1`。

要求：

- 模型列表动态加载。
- 配置实时后台更新。
- 随着支持套餐的渠道而更新。
- 不叠加普通用户分组倍率，避免套餐用户被普通分组价格污染。

套餐消耗公式固定为：

```text
套餐消耗 = 原始模型消耗 × 套餐模型倍率 × 动态倍率(如果该动态倍率规则启用“对套餐生效”)
```

### <个人中心-订阅管理>

订阅管理分为左右两个区域：左侧为套餐订阅，右侧为套餐管理。

#### 1. 左侧：套餐订阅

套餐订阅可以参考钱包管理，但交互需要单独适配。

要求：

- 支持选择套餐。
- 套餐列表使用子区域展示，限制宽高，并支持子项滚动。
- 选择套餐后展示可选支付方式。
- 如果系统配置支持现金支付，需要展示所有可用现金支付渠道。例如现金支付支持微信、支付宝，则这里需要分别展示微信、支付宝。
- 如果配置为余额支付，则只展示余额支付。
- 如果配置为余额&现金都可，则展示余额支付和所有可用现金支付渠道。
- 购买套餐时创建独立订阅订单，不复用充值订单。

#### 2. 右侧：套餐管理

右侧显示用户当前套餐信息，包括：

- 当前套餐名称。
- 订阅周期。
- 套餐总额度。
- 总已用额度。
- 当前刷新窗口额度。
- 当前窗口已用额度。
- 当前窗口剩余额度。
- 下一次刷新时间。
- 套餐状态。
- 是否存在末次超额。

支持操作：

- 续订：生成下一个周期的 `pending` 订阅。
- 升级套餐：立即补差价升级，并重开完整新周期。
- 申请退订/联系客服：只弹窗引导提交工单，不在后端提供实际退订 API。

套餐状态建议至少包括：

- `pending`：待生效。
- `active`：生效中。
- `expired`：已过期。
- `cancelled`：已取消。
- `replaced`：已被升级后的新订阅替换。

### <管理员-套餐管理>

套餐管理使用表格展示，交互可以参考渠道管理。

要求：

- 支持分页。
- 支持新增套餐。
- 支持修改套餐。
- 新增/修改页面参考渠道管理的左右完整菜单交互。
- 列表中支持启用/禁用开关。
- 金额、额度、周期、刷新周期必须由后端校验，前端校验只能作为辅助。

套餐字段建议包括：

- 套餐名称。
- 套餐周期，例如 7 天、1 个月。
- 售卖价格。
- 套餐总额度。
- 刷新周期类型，例如小时、天、月。
- 刷新周期数值，例如每 5 小时刷新一次。
- 每个刷新窗口给予的额度。
- 是否启用。
- 排序。
- 创建时间。
- 更新时间。

#### 刷新额度说明

举例：新建一个 `Lite` 套餐，周期为 1 个月，售卖价格 50 元，套餐总额度为 500。管理员可以配置不是一次性给用户全部 500 额度，而是按刷新周期发放，例如每 5 小时刷新一次，每个 5 小时窗口可用 50 额度。

额度计算规则：

- 每个刷新窗口内，用户最多可使用当前窗口额度。
- 每次 API 消耗都要同时记录总已用额度和当前窗口已用额度。
- 当前窗口结束后，刷新当前窗口额度。
- 总额度始终是上限。
- 如果最后一个刷新窗口理论可用 50，但套餐总剩余额度只剩 20，则该窗口实际最多只能使用 20。
- 前端仍可展示“窗口额度 50”，但可用额度需要按 `min(窗口剩余额度, 总剩余额度)` 计算。
- 如果周期剩余为 0 或总额度已耗尽，即使窗口显示额度，也不能继续调用。

### <渠道管理-编辑/新加>

渠道管理的新建和编辑页面新增一个配置项：是否支持用户订阅套餐调用。

注意：不能复用现有 `ChannelInfo.IsPlan` 或 `is_plan` 字段。现有 `is_plan` 是检测渠道 `base_url` 是否为内置上游套餐地址，语义属于上游供应商套餐；本需求中的字段语义是“该渠道是否允许被用户订阅套餐调用”。

建议新增字段：

```text
support_subscription: bool
```

渠道筛选规则：

- 普通余额用户调用时，仍按现有渠道规则筛选。
- 套餐用户调用时，只能使用 `support_subscription = true` 的渠道。
- 如果当前模型没有任何支持套餐的可用渠道，则返回“无可用渠道”。

### <用户管理-编辑>

用户管理编辑页需要显示当前用户的套餐，并支持管理员操作。

要求：

- 显示当前用户的 `active` 套餐。
- 显示待生效的 `pending` 续订套餐。
- 支持变更套餐。
- 支持移除套餐。
- 支持新加套餐，即提前添加下一个周期的套餐。
- 添加套餐时使用下拉列表选择套餐。
- 添加、变更、移除套餐时需要弹窗二次确认。
- 用户管理中的套餐信息需要和个人中心的订阅管理联动展示。

单活跃套餐规则：

- 同一用户只能存在一个 `active` 订阅。
- 管理员提前添加下个周期套餐时，生成 `pending` 订阅。
- 管理员直接变更当前套餐时，需要明确旧订阅状态处理，例如置为 `replaced` 或 `cancelled`。

### <用户维护-套餐兑换码>

套餐兑换码和普通兑换码的交互基本一致，但需要区分数据结构和字段语义。

要求：

- 入口放在控制台左侧 `用户维护` 下。
- 仅管理员可见，普通用户不可见。
- 可以在运营设置中开启/关闭，默认开启。
- 前端批量生成交互对齐普通兑换码。
- 后端单次限制 100 个，前端通过轮询或分批请求完成更大数量生成。
- 普通兑换码中的额度输入改为套餐选择。

不建议在现有普通兑换码表里简单把 `quota` 换成 `plan_id`，否则会造成字段语义混乱。

建议新建套餐兑换码表，复用普通兑换码的前端批量生成交互和后端 100 条限制。

### <管理员-动态倍率>

动态倍率表单新增配置项：是否对套餐生效。

要求：

- 默认新建动态倍率规则时，不对套餐生效。
- 如果规则启用对套餐生效，则套餐消耗公式中叠加该动态倍率。
- 如果规则未启用对套餐生效，则套餐用户消耗不受该动态倍率影响。

套餐消耗公式：

```text
套餐消耗 = 原始模型消耗 × 套餐模型倍率 × 动态倍率(如果规则启用“对套餐生效”)
```

### <模型广场>

模型广场左侧 `计费类型` 新增 `套餐计费` 筛选项。

要求：

- 不能只是前端新增筛选项，后端也必须返回套餐计费相关信息。
- 模型是否支持套餐计费，需要以后端返回的可用套餐渠道为准。
- 点击模型展开计费详情时，需要新增区域显示套餐计费详情。

后端需要返回的信息建议包括：

- 当前模型是否存在支持套餐的可用渠道。
- 套餐模型倍率。
- 套餐倍率后的价格或消耗。
- 动态倍率是否影响套餐。
- 计算后的最终套餐消耗。

展示建议：

- 使用表格展示套餐消耗倍率和倍率后的价格/消耗。
- 如果模型没有支持套餐的可用渠道，套餐计费区域应明确提示不可用。

## [后端-计费与调用流程]

套餐额度与现有余额扣费模型冲突较大，不能简单扩展用户表字段。当前计费主要围绕用户余额 `quota` 和 `BillingSourceWallet`，而套餐需要处理刷新窗口、总额度、升级、续订和末次超额。

建议新增独立的 `SubscriptionFunding`，与现有 wallet funding 并列。

### API 调用流程建议

1. 判断用户是否存在有效 `active` 订阅套餐。
2. 如果不存在有效套餐，走现有余额计费逻辑。
3. 如果存在有效套餐，进入套餐计费逻辑。
4. 根据模型筛选支持套餐的可用渠道，即 `support_subscription = true`。
5. 如果没有可用渠道，返回“无可用渠道”。
6. 根据套餐模型倍率和动态倍率规则计算预扣额度。
7. 检查当前刷新窗口可用额度和总剩余额度。
8. 如果可用额度不足，直接拒绝，不扣余额。
9. 如果可用额度充足，进行预扣或占用记录。
10. 请求完成后按实际消耗结算。
11. 如果实际消耗高于预扣，且套餐余额不足，允许本次末次超额，写入套餐额度流水，并标记后续不可继续使用套餐额度。
12. 后续请求如果检测到套餐已耗尽或存在不可继续使用的末次超额状态，则直接拒绝。

## [后端-支付回调]

现金支付回调需要特别处理。

现有易支付、Stripe 等回调可能默认完成充值订单。新增套餐订单后，回调必须能识别订单类型，否则会把套餐付款当充值给用户增加余额。

建议：

1. 为套餐订单使用独立 `trade_no` 前缀，例如 `SUB_`。
2. 或者在订单表中维护明确的 `order_type = subscription`。
3. 回调入口根据 `trade_no` 或 `order_type` 查询订单类型。
4. 充值订单进入充值完成逻辑。
5. 套餐订单进入套餐开通、续订或升级逻辑。
6. 回调需要保证幂等，重复回调不能重复开通套餐或重复补差升级。

## [后端-校验要求]

以下内容必须做后端校验，前端校验只能辅助：

- 套餐金额不能为非法值。
- 套餐额度不能为非法值。
- 套餐周期必须合法。
- 刷新周期必须合法。
- 刷新窗口额度不能造成计算异常。
- 用户不能同时存在多个 `active` 订阅。
- 续订只能生成合理的 `pending` 订阅。
- 升级时必须正确保留旧套餐已用额度。
- 升级补差必须按后端可信价格计算。
- 套餐订单支付回调必须幂等。
- 套餐兑换码不能重复兑换。
- 禁用套餐后，不影响已购买且有效的用户订阅，除非另有管理员操作。

请新建一个 CSV 来追踪开发计划，分为五列：

- `顺序`

CSV 中需要覆盖前端、后端、数据库、支付回调、模型广场、渠道管理、用户管理、订阅管理、套餐兑换码、动态倍率、测试与回归等任务。

## [验收重点]

1. 余额支付、现金支付、余额&现金都可三种配置均能正确影响订阅管理页的支付方式展示。
2. 套餐订单和充值订单严格区分，支付回调不会把套餐付款当充值。
3. 用户有有效套餐时，API 调用优先走套餐额度；套餐额度不足时拒绝，不扣余额。
4. 套餐用户只会使用支持套餐调用的渠道。
5. 不复用现有 `ChannelInfo.IsPlan` / `is_plan` 语义，新增独立 `support_subscription` 字段。
6. 用户同一时间只能有一个 `active` 订阅。
7. 续订生成 `pending` 订阅，升级生成新的 `active` 订阅并将旧订阅置为 `replaced`。
8. 升级后旧套餐已用额度正确保留并计入新套餐总额度。
9. 刷新周期按订阅生效时间滚动刷新。
10. 最后一个刷新窗口可用额度不会超过套餐总剩余额度。
11. 实际消耗超过预扣时，允许末次超额并阻止后续套餐调用。
12. 动态倍率只有在启用“对套餐生效”时才影响套餐消耗。
13. 模型广场 `套餐计费` 筛选由后端数据支撑，不只是前端静态筛选。
14. 套餐兑换码和普通兑换码数据表/字段语义清晰区分。
15. 退订按钮只引导提交工单，不让用户误解为已自动退订。
