# 后续补充内容：

## 已确认的关键决策

- API 调用：用户有有效套餐时，所有调用优先走套餐额度；套餐额度不足直接拒绝，不扣余额。
- 渠道标记：渠道是否支持套餐用户调用只用于筛选可用渠道；如果当前模型没有支持套餐的渠道，报“无可用渠道”。
- 用户套餐：单活跃套餐。
- 套餐购买订单：新建独立订阅订单，不复用充值订单。
- 升级：立即补差升级，且重开完整新周期。
- 升级时已用额度：旧套餐已用额度保留到新套餐总额度中。
- 刷新周期：按订阅生效时间滚动刷新。
- 实际消耗超过预扣且套餐余额不足：允许末次超额，记录超额后阻止后续调用。

## 主要质疑 / 缺陷

1. **“套餐”命名和现有项目冲突**
   项目里已有“套餐渠道 / 上游套餐额度查询”概念，例如 `ChannelInfo.IsPlan`、GLM/Kimi/Minimax 套餐查询。这是“上游供应商套餐”，不是“用户订阅套餐”。
   建议内部命名区分：
   - 上游套餐渠道：`provider_plan` / `channel_plan`
   - 用户订阅套餐：`subscription_plan`

2. **“订阅记录和账单合并在一个表”不合理**
   订阅生命周期、支付订单、额度消耗流水是三类不同账务对象，强行合并会导致：
   - 续订、升级、退款、兑换码发放难以表达；
   - API 消耗流水和支付订单混在一起；
   - 后续审计、对账困难。
   建议至少拆成：
   - 套餐定义表；
   - 用户订阅表；
   - 订阅订单表；
   - 套餐额度流水/账单表。
   前端可以“合并展示”，数据库不建议合并。

3. **现金支付回调确实有坑**
   现有易支付、Stripe 回调默认完成充值订单。套餐订单独立后，回调必须能识别订单类型，否则会把套餐付款当充值给用户加余额。
   建议套餐订单使用独立 trade_no 前缀或独立 order_type，并在回调入口统一分发。

4. **套餐额度与现有余额扣费模型冲突较大**
   当前计费主要围绕用户余额 `quota` 和 `BillingSourceWallet`。新增套餐不能简单扩展用户表字段，否则很难处理刷新窗口、总额度、升级、末次超额。
   建议新增独立 `SubscriptionFunding`，和现有 wallet funding 并列。

5. **“支持套餐用户调用”不能复用现有 `ChannelInfo.IsPlan`**
   现有 `is_plan` 是检测渠道 base_url 是否为内置上游套餐地址。你的需求是“该渠道是否允许被用户订阅套餐调用”，语义不同。
   建议新增字段，例如：
   - `support_subscription: bool`
   不要复用 `channel_info.is_plan`。

6. **升级“重开完整周期 + 保留已用额度”需要明确封顶**
   如果旧套餐已用额度大于新套餐总额度，升级后用户应立即不可用，还是允许继续？
   建议：保留已用额度后，如果 `used_total >= new_total_quota`，新套餐立即显示总额度已耗尽，但周期仍有效，不能继续调用。

7. **动态倍率对套餐生效会影响两个价格体系**
   当前动态倍率是叠加在分组倍率上的。套餐又有自己的“模型消耗倍率”。
   建议套餐消耗公式固定为：
   `套餐消耗 = 原始模型消耗 × 套餐模型倍率 × 动态倍率(如果规则启用对套餐生效)`
   不再叠加普通用户分组倍率，避免套餐用户被普通分组价格污染。

8. **模型广场“套餐计费”筛选需要后端支持**
   当前模型广场基于已启用渠道和计费类型展示。套餐计费不能只是前端加筛选项，必须后端返回：
   - 是否存在支持套餐的可用渠道；
   - 套餐模型倍率；
   - 套餐倍率后价格/消耗；
   - 动态倍率是否影响套餐。

9. **兑换码需求要区分普通兑换码和套餐兑换码**
   不建议在现有兑换码表里简单把 `quota` 换成 `plan_id`，否则字段语义混乱。
   建议新建套餐兑换码表，复用前端批量生成交互和 100 条后端限制。

10. **“退订”只弹窗引导工单可以，但后端不应提供实际退订 API**
   如果只是引导工单，UI 上不要出现容易误解的“确认退订成功”按钮。建议按钮文案为“申请退订/联系客服”。

## 建议补充的实现规格

- 套餐状态建议至少包括：`pending`、`active`、`expired`、`cancelled`、`replaced`。
- 用户只能有一个 `active` 订阅；续订生成 `pending`；升级把旧订阅置为 `replaced`，新订阅立即 `active`。
- 套餐额度至少记录：
  - 总额度；
  - 总已用；
  - 当前刷新窗口额度；
  - 当前窗口已用；
  - 下一次刷新时间；
  - 是否末次超额。
- API 预扣先检查当前窗口可用额度；结算时如果实际消耗更高，允许末次超额并写入流水。
- 套餐管理金额、额度、周期、刷新周期都必须后端校验，前端校验只能辅助。
