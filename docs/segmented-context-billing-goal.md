# 多档上下文计费 / 分段计费实现说明

## 参考项目

参考项目已经实现了类似功能，路径为：

`参考项目/new-api`

可以阅读并参考该项目的实现思路、数据结构、前端交互和计费逻辑，但不要直接修改参考项目中的文件。最终实现应落在当前项目中，并保持当前项目既有架构和代码风格。

## 功能名称

后端和业务逻辑中可以称为：

- 多档上下文计费
- 上下文区间计费
- tiered context pricing

前端展示名称统一为：

`分段计费`

## 核心业务规则

该功能不是阶梯拆分计费，而是“按本次请求的输入侧上下文 token 数量命中一个价格区间，然后整次请求使用该区间的价格”。

不要把不同区间拆开分别计费。

举例：

模型 GLM-5.1 有两档价格：

- 32K 以内：输入价格 1，输出价格 3
- 32K 以上：输入价格 2，输出价格 5

如果本次用于区间匹配的上下文 token 数为 20K，则整次请求按 32K 以内价格计费。

如果本次用于区间匹配的上下文 token 数为 50K，则整次请求按 32K 以上价格计费。

不能把 0~32K 按低价、32K~50K 按高价拆分计算。

## token 边界规则

1K 按 1000 tokens 计算。

- 200K = 200000 tokens
- 1000K = 1000000 tokens

区间采用左闭右开规则：

- 0~200K 表示 >=0 且 <200000
- 200K~1000K 表示 >=200000 且 <1000000
- 最后一档可以允许无上限，例如 >=1000000

用于匹配上下文价格区间的 token 数量只看输入侧上下文占用：

```text
context_tokens_for_tier = input_tokens + cache_read_tokens + cache_creation_tokens
~~~

`output_tokens` 不参与上下文区间匹配。

不存在“缓存输出”参与上下文区间匹配的规则。

`output_tokens` 只参与输出费用计算，不参与价格区间命中。

如果当前项目里字段名不是 `input_tokens`、`cache_read_tokens`、`cache_creation_tokens`，请根据现有日志、用量统计和计费逻辑选择语义对应的字段，并在最终总结中说明实际使用了哪些字段。

如果现有字段中某些缓存 token 已经包含在 input tokens 中，不要重复相加。实现前需要先判断字段语义，避免重复计费或错误命中区间。

## 命中区间后的计费规则

命中某个上下文区间后，该区间内配置的多类型价格都应生效，包括但不限于：

- 输入
- 输出
- 缓存创建
- 缓存读取
- 音频输入
- 音频输出
- 图像
- 当前项目已支持的其他计费类型

如果模型或分组未开启分段计费，则必须完全保持原有计费逻辑，不得影响旧模型、旧分组、旧日志和旧价格配置。

## 前端需求一：模型广场

在模型卡片上，如果模型支持分段计费，需要在 `按量计费` 标签右侧增加一个高亮标签。

标签文本为：

```
分段计费
```

标签颜色使用淡蓝色，视觉上要和现有标签风格协调。

在点开的模型卡片详情中，需要展示分段计费价格。

价格表中需要有一列叫做：

```
计费类型
```

对分段计费模型，需要在价格展示中体现不同上下文区间，例如：

- `<200K`
- `≥200K`
- `200K~1000K`

如果同一个上下文区间下有多行不同计费类型，区间/分组单元格应自动合并或视觉上合并，避免重复显示造成表格混乱。

## 前端需求二：系统设置 → 分组与模型定价设置 → 可视化倍率设置

增加一个开关，用于控制当前模型或分组价格是否启用分段上下文计费。

新建配置时，该开关默认关闭。

开关关闭时，保持现有单档价格配置 UI 和数据结构行为。

开关打开时，价格配置 UI 自动切换成多区间配置形式。

开启分段计费后，默认自带两个上下文区间：

- 0~200K
- 200K~1000K

用户可以修改每个区间的上下限。

用户可以新增额外区间。

每个区间都需要覆盖当前项目支持的所有计费类型，不只输入/输出，也包括缓存创建、缓存读取、音频、图像等已有计费类型。

需要做基础校验：

- 区间上下限合法
- 区间不能明显重叠
- 区间配置不能缺少必要价格字段
- 禁止保存明显无效配置

如果当前项目已有价格配置校验逻辑，应优先复用或扩展，不要另起一套重复逻辑。

保存配置时要兼容旧数据结构。不开启分段计费的模型仍使用原有单档配置。

## 前端需求三：消耗日志

每条消耗日志都有日志详情。

当前日志详情中已经记录当前价格、分组倍率、动态倍率费率等信息。

对于启用了多档上下文计费并实际命中某个区间的日志，需要在日志详情中加粗标注：

- 当前为多档计费模式
- 命中的上下文区间
- 用于命中区间的 context_tokens_for_tier
- 命中区间所使用的价格配置

未启用分段计费的日志必须保持原样展示。

日志详情需要能让管理员追溯本次费用为什么命中了某个区间。

## 后端 / 计费逻辑需求

找到当前项目的模型价格配置、分组倍率、动态倍率、消耗日志、token 用量统计、实际费用计算相关代码。

阅读参考项目 `参考项目/new-api` 中对应实现，理解它的数据结构、配置方式、前端展示和计费逻辑。

在不破坏现有逻辑的前提下，新增分段计费配置的存储、读取和计算逻辑。

计费时，如果模型或分组配置启用了分段计费：

1. 先计算本次请求用于区间匹配的上下文长度：

```text
context_tokens_for_tier = input_tokens + cache_read_tokens + cache_creation_tokens
```

1. 根据 context_tokens_for_tier 命中一个上下文价格区间。
2. 命中区间后，整次请求的各类费用都使用该区间内配置的价格。
3. 输出 tokens 只用于输出费用计算，不参与区间匹配。

计费时，如果没有启用分段计费：

- 完全走原有计费逻辑。

分组倍率、动态倍率等当前项目已有逻辑必须继续正确生效。

消耗日志中需要保存足够信息，用于之后查看日志详情时知道：

- 是否为分段计费
- 命中的上下文区间
- 用于区间匹配的 token 数量
- 命中区间的价格
- 计算时使用的分组倍率、动态倍率等倍率信息

避免重复计费、漏计费或错误套用默认价格。

如果需要数据库迁移、配置结构变更或 API 结构变更，请优先参考当前项目已有迁移方式和兼容策略。

不要硬编码具体模型名，比如 GLM-5.1 只作为业务示例。

不要硬编码生产价格。

不要破坏现有单档价格配置。

不要移除旧字段，除非项目已有明确迁移策略。

## 数据结构要求

分段计费配置至少需要表达：

- 是否启用分段计费
- 多个上下文 token 区间
- 每个区间的下限
- 每个区间的上限，可支持无上限
- 每个区间下各计费类型的价格

区间边界应使用整数 token 数存储，不要直接把 `200K` 这种展示文本作为计算依据。

前端展示时可以把 200000 格式化为 200K。

后端计算时必须使用整数 token 数进行比较。

旧配置没有分段计费字段时，默认视为未启用分段计费。

## 验证要求

添加或更新测试，至少覆盖：

- 未启用分段计费时，旧逻辑不变
- 启用分段计费后，context_tokens_for_tier 小于 200000 时命中低区间
- 启用分段计费后，context_tokens_for_tier 大于等于 200000 时命中高区间
- input_tokens 本身小于 200000，但 input_tokens + cache_read_tokens + cache_creation_tokens 大于等于 200000 时，应命中高区间
- output_tokens 很大时，不影响区间命中
- 命中高区间时整次请求都使用高区间价格，而不是拆分计算
- 输入、输出、缓存创建、缓存读取等多计费类型能够按命中区间正确读取价格
- 消耗日志能正确记录分段计费状态、命中区间、context_tokens_for_tier 和命中区间价格
- 前端配置表单能开启/关闭分段计费
- 前端配置表单能新增、修改、保存区间
- 旧配置在没有分段计费字段时仍能正常读取和计费

测试示例：

- input_tokens = 150000，cache_read_tokens = 30000，cache_creation_tokens = 10000，context_tokens_for_tier = 190000，应命中 0~200K。
- input_tokens = 150000，cache_read_tokens = 30000，cache_creation_tokens = 30000，context_tokens_for_tier = 210000，应命中 200K~1000K。
- input_tokens = 190000，output_tokens = 50000，cache_read_tokens = 0，cache_creation_tokens = 0，context_tokens_for_tier = 190000，应仍然命中 0~200K，因为 output_tokens 不参与区间匹配。
- 命中 200K~1000K 后，整次请求都使用该区间价格，不拆分计算。
